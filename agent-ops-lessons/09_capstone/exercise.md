## 练习 1：全机制协同跑通（设计实验类）

七机制不是堆砌，是协同。本练习验证它们的协同关系。

1. **假设**：L01 的诚实收尾路径被 L02/L03 复用；L04 的幂等键是 L05/L06 的地基。
2. **实验设计**：
   - 开 `enable_step_budget` + `enable_cost_budget`，注入「循环诱导 + 预算炸弹」叠加故障。
   - 看哪个机制先触发（步数 vs 成本），触发的都是「诚实收尾」路径。
   - 开 `enable_publish` + `enable_hitl` + `enable_job_registry`，注入「进程崩溃在 publish 后」。
   - 看 L06 续跑时，已执行的 publish 是否被幂等挡住（L04），审批是否免审（L05 first_only）。
3. **预期**：叠加故障下，先触发的机制兜住（步数或成本先到）；续跑时副作用不重放（幂等）、审批免审（幂等重放）。
4. **思考**：为什么这些机制能协同而不冲突？（提示：它们各管不同的失控形态，且复用同一条诚实收尾/幂等地基。）

**验收**：能展示「叠加故障被多个机制协同兜住」的证据，说清协同的依赖关系（L01 路径被复用、L04 幂等是 L05/L06 地基）。

## 练习 3：思考题——这套机制能迁移到别的 Agent 吗（取舍类）

本课的机制都落在 research-assistant，但思路是通用的。

1. **思考**：这套七机制里，哪些是 research-assistant 特有的，哪些是任何循环 Agent 都需要的？（提示：步数/成本/熔断/幂等/审批/恢复/观测——除了「researcher 并行」这个细节，都是通用的。）
2. **思考**：迁移到一个「客服 Agent」（自动回复邮件 + 偶尔触发退款动作）时，哪些机制最关键？（提示：退款是不可重放副作用 → 幂等 + 审批最关键；步数预算次之。）
3. **结论**：用一句话说清这套机制的「可迁移性」（七机制是循环 Agent 的通用生产化清单，不是 research-assistant 专属）。

**验收**：能区分「通用」vs「research-assistant 特有」的机制，并指出客服 Agent 最需要哪些。
